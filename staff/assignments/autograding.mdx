## Empty Submission Detection

Pawtograder automatically detects empty submissions to help identify students who may need assistance. A submission is flagged as empty when:

- The repository contains no files beyond the initial starter code
- No meaningful changes have been made to the starter files
- The submission appears to be unmodified from the template

Empty submissions are flagged in the grading interface and students are notified. This helps instructors:

- Quickly identify students who may be struggling to get started
- Catch accidental submissions before grading
- Provide early intervention for students who need help

Empty submission detection is based on the content of files configured for the assignment, analyzing whether the submission contains substantive changes from the assignment template or starter code.

## Submission File Support

Pawtograder supports both text and binary files in student submissions.

### Text files

Text files (code, configuration, etc.) are displayed with syntax highlighting in the submission viewer. Students and graders can view the file contents directly in the browser.

### Binary files

Binary files (images, PDFs, compiled executables, etc.) are:

- Stored securely in the submission
- Available for download by students and graders
- Displayed with a download button in the file viewer
- Shown with file metadata (name, size, type)

### Markdown preview

Markdown files (`.md`, `.markdown`) are automatically rendered with a live preview, so students and graders can view formatted documentation, README files, and written responses without downloading. The preview supports:

- Standard markdown syntax (headings, lists, links, code blocks)
- Inline code and code blocks with syntax highlighting
- Images and links
- Tables and blockquotes

## Artifacts

Autograders can generate artifacts (files) that are displayed to students and graders alongside submission results.

### Supported artifact types

- **Plaintext files** (`.txt`, `.log`, etc.): Displayed with syntax highlighting and line numbers, useful for test output, logs, or simple reports.
- **Markdown files** (`.md`): Rendered as formatted HTML with proper styling. Ideal for detailed feedback reports or documentation. Supports headings, lists, code blocks, and other markdown features.
- **Binary files** (images, PDFs, etc.): Available for download but not rendered inline.

### Configuring artifacts

To include artifacts in your autograder output:

1. Generate the artifact files in your grading script.
2. Place them in the expected output directory.
3. Configure the artifact list in your assignment settings.
4. Artifacts will automatically appear in the submission view.

### Artifact annotations

Graders can apply rubric checks as annotations directly to artifacts:

- Use `annotation_target: artifact` in your rubric check configuration
- Specify the artifact filename in the `artifact` field
- Graders can select text or sections of the artifact to attach feedback
- Useful for grading written reports, documentation, or generated output files

See the [Rubrics documentation](/staff/assignments/rubrics) for more details on configuring artifact annotations.

## AI-Powered Hints

Pawtograder supports AI-generated hints for autograder test failures. When a student fails a test, they can request a hint that provides guidance without giving away the solution.

### Supported providers

- **OpenAI** — GPT models (default: `gpt-4o-mini`). Use `"provider": "openai"` with models like `gpt-4o-mini` or `gpt-4o`.
- **Azure OpenAI** — Azure-hosted OpenAI models. Use `"provider": "azure"` with your Azure deployment.
- **Anthropic** — Claude models. Use `"provider": "anthropic"` with models like `claude-3-haiku` or `claude-3-sonnet`.
- **OpenRouter** — Access to multiple LLM providers through a unified API. Use `"provider": "openrouter"` with any OpenRouter-supported model (e.g., `openai/gpt-4o-mini`, `anthropic/claude-3-haiku`).

### Configuration

Add an `llm` object to the test's `extra_data` in your test configuration:

```json
{
  "llm": {
    "type": "v1",
    "prompt": "You are a helpful CS tutor. Guide the student to fix their code without giving away the solution.",
    "provider": "openrouter",
    "model": "openai/gpt-4o-mini",
    "account": "default",
    "temperature": 0.85,
    "max_tokens": 500
  }
}
```

Set API keys as environment variables. Use the `account` field in your LLM configuration to select an account-specific key:

- `OPENAI_API_KEY` or `OPENAI_API_KEY_{account}` for OpenAI
- `AZURE_OPENAI_ENDPOINT` and `AZURE_OPENAI_KEY` (or `AZURE_OPENAI_KEY_{account}`) for Azure
- `ANTHROPIC_API_KEY` or `ANTHROPIC_API_KEY_{account}` for Anthropic
- `OPENROUTER_API_KEY` or `OPENROUTER_API_KEY_{account}` for OpenRouter

For OpenRouter, model names use the format `provider/model`, for example:

- `openai/gpt-4o-mini`
- `anthropic/claude-3-haiku`
- `google/gemini-pro`

## AI-Powered Feedback with Feedbot

Feedbot is an AI-powered tool that provides automated feedback on student submissions using large language models (LLMs). It can analyze code quality, provide suggestions, generate personalized feedback at scale, and help students understand their mistakes.

### Supported LLM providers

Feedbot supports multiple LLM providers for flexibility and cost optimization:

- **OpenAI**: GPT-4, GPT-4 Turbo, GPT-3.5 Turbo. Best for high-quality feedback and complex code analysis.
- **Anthropic Claude**: Claude 3 Opus, Sonnet, and Haiku. Best for detailed explanations and educational feedback.
- **OpenRouter**: A single API that provides access to OpenAI, Anthropic, Google Gemini, Meta Llama, and many other models.

**Benefits of OpenRouter:**

- Single API key for multiple providers
- Automatic fallback if primary model is unavailable
- Cost optimization through model selection
- Access to latest models without separate integrations

### Configuring Feedbot

1. Navigate to the assignment settings page.
2. Scroll to the **AI Grading** section and enable **Feedbot**.
3. Select your LLM provider (OpenAI, Anthropic, or OpenRouter).
4. Enter the appropriate API key. For OpenRouter, sign up at [openrouter.ai](https://openrouter.ai) and generate an API key from your dashboard.
5. Choose the model you want to use (e.g., `anthropic/claude-3-sonnet`, `meta-llama/llama-3-70b`).
6. Configure feedback parameters:
   - Feedback style (concise, detailed, encouraging)
   - Focus areas (code quality, style, logic, testing)
   - Temperature (creativity level)
   - Max tokens (response length)
   - System prompt (instructions for the AI)
   - Language and tone preferences

### Feedbot feedback types

- **Code quality analysis**: Identifies code smells and anti-patterns, suggests refactoring opportunities, and evaluates naming conventions and readability.
- **Logic and correctness**: Reviews algorithm efficiency, identifies potential bugs or edge cases, and suggests alternative approaches.
- **Style and best practices**: Checks adherence to language conventions, recommends documentation improvements, and evaluates code organization.
- **Custom prompts**: Create assignment-specific prompts by clicking **Add Custom Prompt** in the Feedbot configuration. Write a prompt describing what to evaluate, specify which files or code sections to analyze, and set priority and weight for this feedback type.

### Viewing Feedbot results

Students see Feedbot feedback on their submission page, including inline comments on specific code lines, summary feedback at the submission level, suggestions for improvement, and links to relevant documentation.

Instructors can review all Feedbot feedback before releasing to students, edit or override AI-generated feedback, disable Feedbot for specific submissions, and track Feedbot usage and costs.

### Best practices

- Start with conservative feedback settings and adjust based on results.
- Review Feedbot output on sample submissions before enabling for all students.
- Combine Feedbot with traditional autograding for comprehensive assessment.
- Use custom prompts to align feedback with course learning objectives.
- Monitor API costs and set usage limits if needed.

## Test Insights and Error Explorer

Pawtograder provides powerful tools to help you understand and address common errors across student submissions.

### Error Explorer

The error explorer groups similar test failures and build errors across all student submissions, making it easy to:

- **Identify patterns**: See which errors affect multiple students.
- **View affected students**: Click on any error group to see the list of students experiencing that issue.
- **Copy student emails**: Quickly copy email addresses of affected students for bulk communication.
- **Pin error groups globally**: Highlight common issues across assignments.
- **Launch regrades**: Start a regrade workflow directly from an error group.

### Regrade workflow

When you need to regrade submissions after fixing a test or updating the autograder:

1. **Launch from error explorer**: Click the regrade button from any error group.
2. **Preselected submissions**: Students affected by that error are automatically selected on the autograder page.
3. **Choose grader version**: Select specific commits from the repository history, or enter a manual SHA in the regrade dialog.
4. **Auto-promote**: Optionally promote the new submission as the active submission if it scores higher.

This streamlined workflow makes it much faster to address issues that affect multiple students, with conflict detection and clearer validation built in.

## Rerunning the Autograder

You can rerun the autograder on existing submissions to apply updated grading logic, fix bugs, or update scores after adjusting test cases. The enhanced rerun feature provides more control over the regrading process.

### Rerun options

- **Reuse the same submission**: The autograder runs again on the existing submission without creating a new one, preserving the original submission timestamp, history, and active-submission status.
- **Select grader version**: Choose which version of the grader to use for the rerun. Use the latest grader commit, select a specific commit SHA, or choose from recent grader versions.
- **Batch operations**: Rerun the autograder on multiple submissions at once.

### How to rerun

You can launch autograder reruns from multiple places:

- **From the assignment page**: Select submissions and click **Rerun Autograder**.
- **From test insights**: Launch reruns directly from error groups. Affected submissions are automatically preselected.
- **From individual submissions**: Rerun the autograder for a single submission.

In the regrade dialog:

1. Choose a grader version from the commit list, or enter a specific commit SHA.
2. Optionally enable **Auto-promote** to set this version as the active grader for all future submissions after the rerun completes successfully.

### Auto-promote

The auto-promote option is useful when:

- You've fixed a bug in the grader and want to apply it going forward
- You've updated test cases and want all new submissions to use them
- You're confident the new grader version should be the default

<Warning>
Rerunning the autograder will replace the existing autograder results for the selected submissions. Make sure you've tested your grader changes before applying them to student submissions.
</Warning>
